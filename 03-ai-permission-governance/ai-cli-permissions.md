# AI CLI Permissions

AI CLI 權限問題的核心不是「AI 會不會」，而是「目前 session 是否被允許讀檔、改檔、執行指令、連外部工具」。

這份文件用來建立權限治理概念。

## 1. 權限類型

| 權限 | 代表意義 | 風險 |
|---|---|---|
| 讀檔 | AI 可以讀取專案內容 | 可能讀到敏感資訊 |
| 改檔 | AI 可以修改檔案 | 可能改錯範圍 |
| 執行指令 | AI 可以跑 build / test / script | 可能執行破壞性命令 |
| 網路 | AI 可以下載或查資料 | 可能引入不可信來源 |
| Git 操作 | AI 可以 stage / commit / push | 可能污染版本歷史 |

## 2. 建議權限策略

### 一般修改任務

允許：

- 讀取專案
- 修改檔案
- 執行測試

禁止：

- 自動 commit
- 自動 push
- 刪除大量檔案
- 修改環境變數與金鑰

### 大型重構任務

額外要求：

- 修改前先列計畫
- 每輪修改後輸出 git diff 摘要
- 必須跑測試或說明不能測的原因
- 建議由第二個 AI 做 diff review

### 生產環境 / 公司專案

預設禁止：

- 直接部署
- 直接改資料庫
- 直接刪資料
- 上傳機敏資訊
- 自動 push 到 main / master

## 3. Claude 常見權限場景

Claude Code 可能在執行指令或修改檔案時要求確認。

遇到確認時，先判斷：

```txt
這個動作是否只影響目前專案？
是否會刪除檔案？
是否會寫入系統層級設定？
是否會連網下載東西？
是否會 commit / push？
```

安全優先順序：

```txt
讀取 < 修改 < 執行測試 < 安裝套件 < 修改系統設定 < commit/push < 部署/刪資料
```

## 4. Codex 常見權限場景

Codex 適合用來：

- 讀取 git diff
- 審查程式碼
- 找風險
- 提供修正建議

Codex 不建議在審查角色中：

- 直接修改檔案
- 自動 commit
- 自動 push
- 自行改掉 Claude 的實作

## 5. 權限提示詞

交給 AI 前可以加：

```txt
你可以讀取與修改目前專案內檔案，也可以執行測試。
但不得自動 commit、不得 push、不得刪除重要檔案、不得修改機敏設定。
如果需要更高權限，請先說明原因與風險。
```

審查型 AI 可以加：

```txt
你只能審查 git diff，不准修改檔案。
請輸出問題、風險、建議修正方向。
```

## 6. 權限異常排查

### AI 一直問權限

可能原因：

- 每次都啟動新 session
- 沒有設定 trust project / allowlist
- 指令超出目前 sandbox
- 嘗試修改系統層級檔案
- 嘗試執行需要 administrator / sudo 的動作

處理：

1. 確認目前在正確專案根目錄。
2. 降低任務範圍，只允許專案內操作。
3. 把高風險動作拆出來人工執行。
4. 讓 AI 只輸出指令，不直接執行。

## 7. 安全底線

永遠不要把以下內容交給公開 repo 或 AI 任務輸出：

- private key
- API token
- production password
- 公司內部 URL / 帳密
- 個資資料集
- `.env` 實際內容

## 8. 結論

AI 權限治理的目標不是把 AI 綁死，而是讓 AI 在明確邊界內加速。

權限越高，驗證與審查成本越高。
